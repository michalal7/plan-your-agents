# Shared analysis machinery (setup-dev-agents & setup-task-agents)

Both agent-setup skills share this file. Each `SKILL.md` stays thin and points here for
the parts that don't differ: how to load the knowledge base, how to scan a repo, the mode
logic, and how to write the plan file. The skill-specific *lens* (which questions, which KB
sections, what the plan recommends) lives in each `SKILL.md`.

Read this file once at the start of a run, together with your skill's own `SKILL.md`.

---

## 1. Locate the knowledge base
Read the KB before deciding anything. Use the **first** path that exists:
1. `knowledge/claude-agents/` bundled next to the running `SKILL.md` (the plugin case — a
   generated mirror; you are given this skill's base directory at launch, so the absolute
   path is `<skill-base-dir>/knowledge/claude-agents/`),
2. `~/.claude/knowledge/claude-agents/` (global standalone install),
3. `.claude/knowledge/claude-agents/` in the target repo (repo-local).

Start with `PLAYBOOK-agent-design.md`; pull topic files `00`–`90` and `INDEX.md` as needed.
The KB is authoritative — claim nothing from memory about flags/settings/modes. Heed ⚠️
markings and `90-deprecated.md`.

## 2. Determine mode and target from the argument
`<arg>` = target repo (empty or `.` = current directory). Mode = `apply` if the argument
contains the word `apply`, otherwise `advise` (default).
- **advise** (default, read-only for the repo): scan, decide, and **write the plan file**
  (§4). Change nothing else.
- **apply**: everything advise does, **plus** scaffold the recommended files after the user
  confirms. Never overwrite an existing file without asking. The plan file itself is a
  generated output — overwriting it on a re-run is fine.

## 3. Scan the repo (read-only)
Common signals both skills need:
- language/stack; VCS; repo size / monorepo;
- test/build/lint commands (package.json scripts, Makefile, pyproject, …);
- existing `.claude/` (CLAUDE.md, settings.json, agents, commands, skills);
- domain (backend / frontend / mobile / data / non-code).

**If a plan file already exists at the target path, read it before scanning.** It is the
only source for the Verdict's re-run delta line — without it that line would be recalled
rather than read, which §5 forbids.

Then gather the signals your own `SKILL.md` calls for (dev-cycle specifics vs. the repo's
workload/goal). Recognize **non-code repos** (notes/docs/vault: PDFs, MD, no `src`/tests) and
don't force a code setup on them.

## 4. Write the plan file
Every run writes a markdown plan to the **project root** (overwrite if it exists):
- `setup-dev-agents` → `agent-dev-plan.md`
- `setup-task-agents` → `agent-task-plan.md`

Use this skeleton; fill each section from your skill's lens, keep it directive and concrete
(no filler — every line should influence a decision):

```markdown
# Agent plan — <dev | task> — <repo name>
_Generated by /<skill-name> on <date>. Mode: <advise | apply>. Read-only analysis unless apply._

## Verdict
The answer, before the reasoning: one sentence on what to do — including "nothing needs
changing" when that is the honest result. Then an effort class (**minutes / hours / days**)
for a reader who cannot estimate the cost themselves. The class is a declared estimate, not
a measured fact — label it as such, and it is exempt from the read-it-first rule in §5.
Re-run only: add **one** line for what changed since the last plan.

## 1. What's there
Repo, stack, domain, size; the signals found (test/build/lint commands, existing `.claude/`
config, workload/goal). Facts only — no recommendations yet.

## 2. Decision
The recommendation + rationale. Cite the decision (PLAYBOOK §1 orchestration level for
task; §2/§3 for dev) and answer the two guiding questions where relevant
(workers talk? repeatable/scriptable?).

## 3. Verification loop
The mandatory check for this repo (KB principle #1). No loop → no recommendation.

## 4. Setup / architecture
Concrete files and contents (dev: CLAUDE.md / settings / agents / commands; task:
orchestration + pattern + verification, plus any workflow/command/agent stubs). For each
item, say what deliberately should NOT go in and which PLAYBOOK §5 trap that avoids —
stated with the item, not collected into a section of its own.

Close the section with a one-line sweep of the PLAYBOOK §5 traps that attach to the plan as
a whole rather than to any single item — missing feedback loop, forced Plan Mode, agents on
a task that doesn't decompose, scaffolding that outlives its reason, over-broad permissions,
re-prompting instead of writing down. **Run this sweep even when the Verdict is "nothing
needs changing"** — a plan that recommends nothing still has to show it checked.

## 5. Next steps
What apply would create; what the user must still check.
```

Sections stay in this order and none is dropped. When the Verdict is "nothing needs
changing", §2–§4 say what is already right and why, briefly — they do not disappear.

**Each recommendation carries one additional line, written in plain language** — no jargon,
no §-references, no tool names the reader would have to look up. The technical wording stays
as it is; this is a companion line, not a replacement. A plan is read by people who decide
about the work as well as by those who do it, and a recommendation nobody outside the team
can evaluate is not actionable.

After writing, tell the user the plan's path and give a 3–5 line summary in chat — don't
paste the whole file back.

## 5. Rules (both skills)
- The verification loop is mandatory in every recommendation (KB principle #1).
- Adapt to the concrete repo; never run a template off verbatim.
- Claim nothing safety-relevant without KB/docs coverage.
- **Every number and every attribution must be read in this run, not recalled.** Counts,
  file sizes, version strings, and any "file X says Y" go into the plan only if you opened
  the file during this run. If you did not, leave it out or say it is unchecked — a precise-
  sounding figure is worse than no figure, because it cannot be told apart from a verified
  one. (This is broader than the safety rule above on purpose: a wrong count discredits a
  plan the same way a wrong flag does.)
- **Keep the plan under ~120 lines** (prose lines; headings and fenced blocks don't count).
  It is a decision document, not a record of the run: resist retelling what changed on a
  re-run — the Verdict's one delta line covers it and the repo's history holds the rest.
- advise never edits repo files other than the plan file; apply scaffolds only after
  explicit confirmation and never overwrites without asking.
